# Данные для LLM-рекомендации

Backend передаёт модели текущую потребность продавца и компактный агрегат его
состояния. Сырые строки БД, персональные данные и история запросов не передаются.

## Формат

Данные отправляются в Ollama как компактная JSON-строка в единственном текущем
`user`-сообщении:

```json
{
  "need": "Хочу увеличить средний чек за счёт сопутствующих товаров",
  "state": {}
}
```

Каждый вызов является one-shot: предыдущие запросы и ответы не добавляются.

## Состав `state`

| Блок | Что передаём | Как рассчитываем |
| --- | --- | --- |
| `inventory` | Количество активных товаров и позиций, наличие, общий и типичный остаток, плохую оборачиваемость, КИЗ | По активным товарам продавца и их активным `product_items`; остатки суммируются сначала на уровне карточки |
| `pricing` | Минимальную, среднюю, медианную и максимальную цену, базовые цены, скидки и долю товаров со скидкой | Эффективная цена: `club_discounted_price`, затем `discounted_price`, затем `price` |
| `assortment` | Количество категорий, типов и брендов, краткие сводки и примеры товаров | Группировка активных товаров по `parent_id`, `subject_id` и бренду |
| `usage` | Количество активных, будущих и завершённых промокодов, комплектов и участий в акциях, охват и пересечения товаров | Статусы вычисляются по текущей дате `Europe/Moscow`; учитываются только данные текущего продавца |
| `ready` | Техническую готовность к промокодам и комплектам | Промокод доступен при наличии хотя бы одного товара; комплект — минимум двух товаров в наличии |
| `market` | Доступные акции маркетплейса и соответствие продавца требованиям | Проверяются сроки вступления, количество товаров, остатки и допустимые категории |

Дополнительно передаются:

- `at` — время формирования контекста в UTC;
- `date` — текущая бизнес-дата в `Europe/Moscow`;
- `currencies` — валюты активных товарных позиций.

## Ограничения объёма

- категории — максимум 3;
- типы товаров — максимум 3;
- примеры товаров — максимум 3;
- доступная акция — максимум 1;
- категории внутри акции — максимум 3;
- преимущества акции — максимум 2;
- весь JSON пользовательского сообщения — максимум 2600 символов.

При превышении лимита backend последовательно сокращает примеры товаров, типы и
категории. Скалярные агрегаты, признаки готовности и лучшая акция сохраняются.

## Что не передаём

- username, email, имя продавца, `seller_sid` и данные авторизации;
- ID, артикулы, штрихкоды, описания и фотографии товаров;
- сырые списки всех товаров и позиций;
- историю диалога;
- продажи, выручку, маржу, конверсию и аудиторию — этих данных сейчас нет в БД.

До сборки `state` backend проверяет, что `need` содержит явную цель продаж или
продвижения. Посторонний ввод получает `clarification_required` без вызова LLM.
